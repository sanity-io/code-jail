# jail — Functional Specification (UX & Behavior)

## 1) Purpose & Principles

* **Purpose:** Make it trivial to spin up a safe, disposable **devcontainer “jail”** for running coding agents with “safety off,” while keeping host machine guarded and setup friction near zero.
* **Design tenets:**

  * **Predictable:** One command, same behavior everywhere (macOS/Linux; Docker or Podman).
  * **Composable:** Works in terminals, CI, and editors. No lock-in to VS Code; devcontainer spec is standard.
  * **Opinionated defaults, configurable later:** Always include a coding agent (**codex**) inside the container and wire it for “dangerous” mode when explicitly requested.
  * **Clear risk boundaries:** “Dangerous” implies file/network/process authority **inside** the jail only.

## 2) Supported Environments (initial)

* **Host OS:** macOS, Linux.
* **Container runtime:** Docker (required initially). (Podman: later.)
* **Dependencies (host):** `docker`, `codex` (installed and authenticated on host), shell (bash/zsh), and basic POSIX tools.
* **Project layout:** Operates in the current working directory (project root).

---

## 3) Top-Level UX & Commands

### 3.1 `jail` (no args)

**Behavior:**

* If `.devcontainer/` exists in CWD:

  * Starts the devcontainer and **drops the user into an interactive shell inside the container**.
  * Uses the devcontainer.json present (generated by `jail init` or hand-rolled).
* If **no** `.devcontainer/`:

  * Prints: *“No devcontainer found in this repo. Run `jail init` to create one.”*
  * Exits with non-zero.

**Common flags (apply to `jail`, `jail init`, `jail codex`):**

* `-v, --verbose` – verbose logs.
* `--dry-run` – print actions (no changes).
* `--runtime=docker|podman` – override runtime detection (default: docker).
* `--non-interactive` – fail if a prompt would appear; require all inputs via flags/env.

---

### 3.2 `jail --help`

**Behavior:**

* Prints concise CLI help with commands, flags, examples.
* Always exits zero.

**Help content (example excerpt):**

```
Usage:
  jail [--verbose] [--runtime docker]        # open shell in existing jail
  jail init ["platform requirements"] [flags]
  jail codex [codex args...]
  jail --help | -h

Examples:
  jail init "a python cli"
  jail            # open shell if .devcontainer exists
  jail codex exec "build the repo"
```

---

### 3.3 `jail init`

**Purpose:** Create a ready-to-run devcontainer in the current folder and adapt it to the project via a Codex instruction.

**Inputs:**

* `jail init "<high-level description>"` — provide a single free‑text prompt (preferred).
* `jail init` — interactive: asks for one high‑level description (one or two sentences).

**Behavior:**

1. **Bootstrap template** into `.devcontainer/` if absent:

   * `devcontainer.json` (minimal, opinionated base; includes codex install step).
   * `Dockerfile` (base image + codex deps).
   * `postCreateCommand.sh` (optional provisioning).
   * `.devcontainer/jail.config.json` (jail metadata; see config).
2. **Gather requirements**:

   * Use the provided prompt or collect a single high‑level description interactively.
3. **Adapt via Codex (host):**

   * Compose a **Codex instruction** from the user inputs (or CLI prompt).
   * Run **host codex** to **edit** or **generate** the container spec files to align with requirements (e.g., add apt packages, install tools, set default shell, etc.).
   * **Always ensure codex is installed inside the container** (via Dockerfile or postCreate).
4. **Validate:** lint resulting `devcontainer.json`, check Dockerfile builds, and that `codex` is present in final image.
5. **Report & next steps:**

   * Print: “Devcontainer created. Run `jail` to enter the jail.”

**Flags:**

* `--image <name[:tag]>` – start from a base image (e.g., `mcr.microsoft.com/devcontainers/base:ubuntu`).
* `--node <ver>`, `--python <ver>`, `--go <ver>`, etc. — quick version pins (optional conveniences).
* `--no-ai-on-init` — skip AI adaptation during init; just drop a vanilla template.
* `--overwrite` — overwrite existing `.devcontainer/` (default: refuse if exists).
* `--name <label>` — set container name prefix (defaults to folder name).

**Interactive flow (example):**

* Prompt: “Describe at a high level what you plan to do (one or two sentences).”
* Run Codex adaptation → show plan/logs (if `--dry-run`, only show planned writes).

**Outputs:**

* `.devcontainer/` directory ready to use.
* Minimal logs on success; verbose logs if `--verbose`.

---

### 3.4 `jail codex [codex args…]`

**Purpose:** Execute **codex from within the container** with host config propagated, designed for “safety off” workflows (explicit).

**Behavior:**

* If `.devcontainer/` missing → instruct to run `jail init`.
* Ensures devcontainer is running (start if needed).
* **Runs** `codex` **inside the container** with arguments passed through.
* codex runs with `--dangerously-bypass-approvals-and-sandbox` and `--skip-git-repo-check`.

* **Propagates host `~/.codex`** (ephemeral copy) to authenticate.

  * Only codex credentials are propagated; no other host secrets by default.

**Flags (besides pass-through):**

* `--config <path>` – override which host codex config to propagate (default: `~/.codex`).
* `--no-propagate-config` – do not mount/copy host config; then codex inside uses its own (likely unauthenticated).
* `--workdir <path>` – working directory inside container (default: `/workspaces/<repo>` equivalent).

**Examples:**

* `jail codex` → runs `codex --dangerous` interactively inside jail.
* `jail codex run "set up the project"` → passes through to codex subcommand.
* `jail codex --on-request run "build and test"` → overrides default dangerous.

## 4) Container & Filesystem Model

* **Mounts:**

  * Project root mounted into container at `/workspaces/<repo>` (read-write).
  * Optional: mount `~/.codex` into container at `/home/dev/.codex` (read-only) unless `--no-propagate-config`.
* **User mapping:** Container user should match a non-root user (UID/GID mapping) to avoid root-owned files on host.
* **Networking:** Default allow outbound networking; no inbound ports unless user runs servers. (Port forwards handled by Docker default behavior; no special mapping unless devcontainer.json specifies.)
* **Editor integration:** Generated devcontainer is standard; VS Code and others should “just work” if launched.

## 5) Templates (initial set)

**Base template contents (minimal):**

* `.devcontainer/devcontainer.json` (example baseline)

  ```json
  {
    "name": "jail",
    "build": { "dockerfile": "Dockerfile" },
    "remoteUser": "dev",
    "workspaceFolder": "/workspaces/${localWorkspaceFolderBasename}",
    "mounts": [
      "source=${localEnv:HOME}/.codex,target=/home/dev/.codex,type=bind,consistency=cached,readonly"
    ],
    "postCreateCommand": "./postCreateCommand.sh"
  }
  ```
* `.devcontainer/Dockerfile` (baseline)

  ```Dockerfile
  FROM mcr.microsoft.com/devcontainers/base:ubuntu
  # Create non-root user
  RUN useradd -m -s /bin/bash dev && usermod -aG sudo dev && echo "dev ALL=(ALL) NOPASSWD:ALL" >> /etc/sudoers
  USER dev
  WORKDIR /workspaces
  # Install codex inside container (placeholder; codex team specifics)
  RUN curl -fsSL https://get.codex.sh | bash || true
  ```
* `.devcontainer/postCreateCommand.sh`

  ```bash
  #!/usr/bin/env bash
  set -euo pipefail
  echo "postCreate: ready"
  ```
* `.devcontainer/jail.config.json` (metadata for future automation)

  ```json
  {
    "version": 1,
    "adaptedBy": "codex",
    "lastAdaptPrompt": "",
    "imageBase": "mcr.microsoft.com/devcontainers/base:ubuntu"
  }
  ```

**Codex adaptation step modifies:**

* Add language runtimes (e.g., Python 3.11, Node 22) and package managers.
* Add OS packages (build-essential, git, curl, jq, etc.).
* Add useful dev CLIs (uv/poetry/pipx; pnpm/yarn; cargo tools).
* Optionally tweak `postCreateCommand.sh`.

## 6) Configuration & Conventions

### 6.1 `jail.config.json` (owned by jail)

* **Purpose:** Track adaptation & choices; never user-edited unless advanced.
* **Fields:**

  * `version`: number
  * `adaptedBy`: `"codex"|...`
  * `lastAdaptPrompt`: string
  * `imageBase`: string
  * `runtimes`: `{ "python": "3.11", "node": "22", ... }` (optional)
  * `tools`: `["uv","poetry","pnpm"]` (optional)

### 6.2 Non-interactive operation

* `--non-interactive` requires a prompt string or sufficient flags to avoid Q\&A.
* If inputs are insufficient, exit with error explaining missing bits.

---

## 7) Safety & Trust Model

* **Scope of “dangerous”:** When run with `jail codex --dangerous`, elevated/destructive operations are permitted **inside the container only**. The host is protected by:

  * Limited mounts (project folder, optional `~/.codex` read-only).
  * Non-root container user by default.
* **Warning UX:**

  * First-time **per repo**: print a clear, one-line warning when `--dangerous` is in effect:
    *“Running codex in dangerous mode inside the jail. Host is not directly writable except mounted workspace.”*
  * Provide `JAIL_SUPPRESS_WARN=1` to silence.

---

## 8) Errors, Exit Codes, and Logging

* **Exit codes:**

  * `0` success
  * `1` generic failure
  * `2` missing dependency (docker, codex)
  * `3` invalid devcontainer (lint/build failed)
  * `4` non-interactive input missing
* **Error messages:** Single-line summary + hint to fix; verbose mode prints stack/trace and the docker build logs.
* **Validation steps:**

  * On `init`, validate `devcontainer.json` schema and attempt a **quick build** (`--dry-run` prints planned Docker build args and file diffs).
  * On `jail`, verify container can start; report if port conflicts or mount failures occur.

---

## 9) Telemetry (optional, off by default)

* If enabled via `JAIL_TELEMETRY=1`, collect anonymous counters:

  * `init` runs, language selections, build success/fail.
* Never collect project names or prompts; only coarse stats.

## 10) Examples

**Create for a Python CLI quickly (non-interactive):**

```bash
jail init "a python cli using uv and pytest; needs curl and jq"
jail        # enter the jail
jail codex  # runs codex --dangerous inside the jail
```

**Create interactively for Node/TS:**

```bash
jail init
# choose: Node/TS
# prompt: "Node 22, pnpm, playwright, needs chromium deps"
# → adapts Dockerfile & postCreate via codex
```

**Run codex with safer policy:**

```bash
jail codex --on-request run "scaffold tests"
```

**Force overwrite the devcontainer boilerplate:**

```bash
jail init "rust workspace with cargo-nextest" --overwrite
```

**Dry run to preview changes:**

```bash
jail init "go 1.23, goreleaser, delve" --dry-run --verbose
```

## 12) Future-proofing Hooks

* **Agent abstraction:** Reserve subcommand pattern:

  * `jail <agent> [args…]` e.g., `jail mildred …`, `jail deep …`
* **Profiles:** `jail init --profile <name>` for curated stacks (web, data-science, rust-cli).
* **Service add-ons:** `jail add postgres`, `jail add redis` to append dev services.

---

## 13) Acceptance Criteria (MVP)

* `jail --help` prints usable help.
* `jail` opens a shell when `.devcontainer/` exists; otherwise suggests `jail init`.
* `jail init` (interactive and prompt string):

  * Creates `.devcontainer/` with working `devcontainer.json` + `Dockerfile`.
  * Runs a host codex adaptation step that edits files according to requirements.
  * Ensures `codex` is installed inside container.
  * Validates that the container builds successfully (or fails with actionable error).
* `jail codex`:

  * Starts container if needed.
  * Runs `codex` **inside** container.
  * Defaults to `--dangerous` when no policy flag is provided.
  * Propagates host `~/.codex` read-only by default (can be disabled).
